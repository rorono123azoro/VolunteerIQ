2. Problem Statement
Volunteer coordination in NGOs and nonprofits is largely manual, fragmented, and data-blind. Coordinators rely on spreadsheets, WhatsApp groups, and phone calls to manage thousands of volunteers. This results in:

Skill-opportunity mismatches — volunteers are assigned tasks that don't align with their expertise or interests, causing frustration and dropout.
No real-time visibility — coordinators have no live dashboard to see who is available, where they are, or which tasks are critically understaffed.
Zero impact measurement — NGOs cannot quantify their social impact in a credible, data-backed way, limiting their ability to attract donors and grants.
High volunteer dropout rates — without engagement nudges and recognition systems, volunteers lose motivation and stop participating. Average dropout is approximately 40%.
Inefficient scheduling — there is no system to predict future volunteer demand or optimize schedules based on historical attendance patterns.

Target SDGs:
SDG 17 — Partnerships for the Goals: VolunteerIQ strengthens multi-stakeholder partnerships between NGOs and individual volunteers.
SDG 10 — Reduced Inequalities: By democratizing access to volunteer coordination tools, VolunteerIQ enables smaller NGOs with limited resources to operate as efficiently as large organizations.

3. Product Vision & Goals
VolunteerIQ transforms volunteer management from a manual, reactive process into an intelligent, data-driven system that proactively matches the right people to the right opportunities, tracks participation in real time, and automatically measures and communicates social impact.

3.1 Primary Goals
Reduce volunteer dropout rate by 35% through smart matching and engagement nudges.
Save NGO coordinators at least 5 hours per week by automating scheduling, notifications, and reporting.
Achieve 85%+ volunteer-to-opportunity matching accuracy using Gemini API embeddings.
Enable NGOs to generate automated, credible social impact reports shareable with donors.
Provide a deployable, scalable platform using Firebase that works for NGOs of any size.

5. UI / UX Design Requirements
Design Philosophy: Minimalistic and Classic
VolunteerIQ's interface must feel effortless and trustworthy. The visual language is deliberately restrained — clean white space, subtle borders, neutral typography, and purposeful use of a two-color accent palette. No gradients, no animations, no decorative elements. Every pixel serves a function. At the same time, the layout follows classic web conventions: clear hierarchy, familiar navigation patterns, and structured data tables — so that first-time users feel immediately oriented.

5.1 Design Principles
Whitespace first — generous padding and margins. Content breathes; pages never feel cluttered.
Typography hierarchy — three clear levels: page title (28px bold), section heading (18px semi-bold), body (14px regular). Font: Inter or system-ui sans-serif.
Two-color accent system — Navy (#1B2A4A) for primary actions and headings. Forest Green (#2D6A4F) for success states, badges, and CTAs. All other UI elements in neutral grays.
Classic layout grid — 12-column grid. Sidebar navigation (240px fixed) + main content area. Standard card-based component system.
No dark mode in MVP — single light theme only. Reduces scope and ensures consistent judge experience.
Mobile-responsive — all pages must work on 360px screens. Priority layout: stacked single-column on mobile.
Data-forward design — dashboards and tables are the hero elements. Charts are simple: bar, line, donut — no 3D, no animated charts.

6.2 Feature Specifications
F-01: Firebase Authentication
Implement Google Sign-In via Firebase Authentication SDK.
On first login, detect if user is a Volunteer or NGO Coordinator (role selection screen).
Store user role in Firestore users collection.
All protected routes require Firebase Auth token verification.
Session persistence: users stay logged in across browser refreshes.

F-02: AI Volunteer Matching (Gemini API)
On profile creation, send the volunteer's skills list to Gemini API via the text-embedding-004 model.
Store the resulting embedding vector in Firestore under the volunteer's document.
When an NGO posts an opportunity, generate an embedding for the required skills.
At query time, compute cosine similarity between a volunteer's embedding and all open opportunities.
Apply a location penalty factor: reduce score by 0.1 for each 10km beyond the volunteer's preferred radius.
Final match score = (0.70 × skill similarity) + (0.20 × location score) + (0.10 × availability overlap).
Display final score as a percentage badge on each opportunity card.

F-03: Real-Time Coordination (Firebase Realtime DB)
Event check-in: volunteers tap 'Check In' on the event detail page. Status written to Firebase Realtime DB immediately.
Coordinator dashboard listens to Firebase real-time updates — no page refresh needed.
Status transitions: Registered → Checked In → Completed / No-Show.
Coordinator can manually override status for any volunteer.
Push notifications via Firebase Cloud Messaging (FCM) when a new opportunity matches a volunteer's profile, when a coordinator sends an alert, or when an event is starting in 30 minutes.

F-04: Auto-Generated Impact Report (Gemini API)
Coordinator clicks 'Generate Impact Report' for a date range.
Backend queries Firestore for all completed events, attendance records, and logged hours.
Aggregated stats are passed to Gemini API with the prompt: 'Write a professional, empathetic 3-paragraph impact summary for an NGO coordinator. Data: [JSON summary]. Include specific numbers. Tone: warm and credible.'
Response is rendered in the Impact Report screen along with computed charts.
PDF export via browser print CSS (print-friendly layout applied automatically).

7. Database Design — Firebase Firestore
All persistent data is stored in Firebase Firestore (Cloud Firestore), a serverless, NoSQL document database.
Real-time attendance and presence data is stored in Firebase Realtime Database for low-latency updates.
Firestore security rules enforce that users can only read/write their own documents unless they hold a coordinator or admin role.
All timestamps are stored as Firestore Timestamps (UTC). All IDs are auto-generated Firestore document IDs unless otherwise specified.

7.9 Firebase Realtime Database — Live Presence
Path structure: /live_events/{eventId}/volunteers/{volunteerId}
Data stored: { status: 'checked_in', lastSeen: <timestamp>, location: { lat, lng } }
The coordinator dashboard subscribes to /live_events/{eventId} with an onValue() listener.
This gives sub-second updates when a volunteer checks in, without polling.
Data is ephemeral — cleared after event completion. Not used for permanent storage.

7.10 Firestore Security Rules Overview

The following rules enforce role-based access control:
Authenticated users can read and update their own /users/{uid} document.
Volunteers can read all /opportunities where status == 'open'.
Volunteers can create /applications for themselves. Cannot read other applications.
Coordinators (role == 'coordinator') can read/write all documents under their organizationId.
Impact reports are readable only by coordinators of the owning organization.
Admin role has full read/write access to all collections.
All writes are rejected if the user is not authenticated (Firebase Auth required).

8. System Architecture
8.1 Architecture Overview

VolunteerIQ uses a serverless, event-driven architecture. The frontend is a React.js SPA deployed on Firebase Hosting. The backend is a FastAPI (Python) application deployed on Google Cloud Run — it handles AI matching, report generation, and complex queries. Firebase Firestore and Firebase Realtime DB handle all persistent and real-time data. Firebase Authentication manages identity. All AI calls are proxied through the backend to keep API keys secure.

8.2 Data Flow — Volunteer Matching

Volunteer completes profile and submits skills list.
Frontend sends skills array to Cloud Run backend via REST API.
Backend calls Gemini API (text-embedding-004) to generate a 768-dim embedding for the skills.
Embedding is stored in Firestore under /users/{uid}/skillEmbedding.
When volunteer loads the opportunity feed, backend fetches all open opportunities with their embeddings.
Backend computes cosine similarity between volunteer embedding and each opportunity embedding.
Backend applies location and availability scoring factors.
Ranked list of opportunities with match scores is returned to the frontend.
Frontend renders opportunity cards in ranked order with match score badges.

8.3 Data Flow — Real-Time Attendance

Event is created by coordinator in Firestore (/events collection).
Accepted volunteers see the event in their dashboard with a 'Check In' button.
On check-in, frontend writes to Firebase Realtime DB at /live_events/{eventId}/volunteers/{uid}.
Coordinator dashboard has an active onValue() listener on /live_events/{eventId}.
Dashboard updates instantly (< 500ms) showing the volunteer as checked in on map and table.
On checkout, attendance record is finalized in Firestore /attendance collection.
Cloud Function triggers to update volunteer's totalHours in /users/{uid}.

9. End-to-End Workflow
9.1 Volunteer Workflow

Volunteer visits volunteeriq.web.app on any device.
Clicks 'Continue with Google' — Firebase Auth handles sign-in.
First-time user: completes 3-step onboarding (details → skills → availability).
System generates Gemini embedding of skills and stores it in Firestore.
Volunteer lands on their personalized dashboard.
Opportunity feed displays ranked cards, each showing organization, task, date, and match score.
Volunteer clicks an opportunity → opens detail page with full description, map, and NGO profile.
Clicks 'Apply Now' → application document created in /applications.
Coordinator receives push notification of new application.
Coordinator reviews and accepts → volunteer receives FCM notification: 'You're confirmed!'
On event day, volunteer opens the app and taps 'Check In'.
Coordinator dashboard shows the volunteer as present in real time.
After event, volunteer taps 'Check Out'. Hours are logged automatically.
Volunteer dashboard updates: total hours and events attended increase.

9.2 NGO Coordinator Workflow

Coordinator logs in with Google. Role detected from Firestore — redirected to coordinator portal.
Creates an organization profile (first time) or accesses existing dashboard.
Posts a new opportunity: title, description, required skills, date, location, slots needed.
System generates Gemini embedding for required skills. Opportunity status set to 'open'.
Opportunity appears in volunteer feeds ranked by match score.
Coordinator receives application notifications. Reviews applicants in the roster tab.
Accepts or rejects applicants. Accepted volunteers are moved to confirmed list for the event.
On event day: opens 'Live View' — sees real-time map of checked-in volunteers.
Can send push alerts ('Event starts in 10 minutes!') from the dashboard.
After event ends: marks event as 'Completed'. Hours finalized in Firestore via Cloud Function.
Generates Impact Report for the month — Gemini writes the narrative summary automatically.
Shares report PDF link with donors or copies the impact card to social media.

9.3 System Automation (Cloud Functions)

Trigger: onWrite /attendance/{id} — when status changes to 'completed', update user's totalHours and eventsAttended in /users/{uid}.
Trigger: onWrite /opportunities/{id} — when volunteersRegistered reaches volunteersNeeded, auto-update status to 'full'.
Trigger: Scheduled (daily 8:00 AM) — find events starting within 24 hours, send FCM reminders to all confirmed volunteers.
Trigger: Scheduled (weekly Sunday) — compute week-over-week stats for each organization and write to Firestore analytics subcollection.

11. Non-Functional Requirements
11.1 Performance
Opportunity feed must load with match scores in under 2 seconds for up to 100 opportunities.
Real-time attendance updates must reflect in the coordinator dashboard within 500ms.
Firebase Hosting CDN must serve all static assets with a Time to First Byte (TTFB) < 200ms.
Gemini API calls must complete within 3 seconds. Implement timeout and fallback messaging.

11.2 Scalability
Firestore scales horizontally without manual intervention. Design queries to avoid collection-group scans.
Cloud Run autoscales backend instances based on request load. Set min-instances: 0 for cost efficiency.
All opportunity queries must use Firestore composite indexes (city + status + eventDate) to prevent full scans.

11.3 Security
All Firebase Firestore reads/writes protected by security rules — no open collections.
Gemini API keys and Firebase service account credentials stored as Google Cloud Secret Manager secrets.
Backend validates Firebase ID token on every request using firebase-admin SDK.
No sensitive data (API keys, service accounts) committed to GitHub repository.

11.4 Accessibility
All interactive elements have visible focus indicators.
Minimum contrast ratio 4.5:1 for all body text (WCAG AA compliance).
All images and icons have descriptive alt text.
Form fields have associated labels — no placeholder-only identification.

11.5 Browser Support
Google Chrome (latest 2 versions) — primary target.
Firefox and Safari (latest) — supported.
Mobile Chrome on Android 10+ — fully supported.
Internet Explorer — not supported.
